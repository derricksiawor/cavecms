# Plan 09 (Deployment / Ops) — Definition of Done verification

**Status as of cluster-6 ship (commit `04042be`, 2026-05-14).**

This document tracks each Definition-of-Done item from Plan 09's
Task 13 (`docs/superpowers/plans/2026-05-10-cavecms-09-deployment-ops.md`
lines 1537-1548) and classifies it:

- **✅ Verified-in-code** — the artifact exists, has been audited, and
  is correct. The next step is operator execution; no further code
  work is required.
- **🟡 Operator-action** — requires a human to run something on the
  actual server, or perform a cutover ritual. Cannot be automated by
  code in this repo.
- **⚠️ Verifiable-only-in-prod** — can only be confirmed once a real
  production deploy has happened (e.g. "rollback works on a broken
  release"). Code is in place to support the check.

## 1. `scripts/setup.sh` succeeds + is idempotent

**Status:** ✅ verified-in-code · 🟡 operator-action

`scripts/setup.sh` was shipped in cluster-1 and hardened through 6
audit rounds (final cluster-3 commit `42cff59`). Idempotency is built
in via the standard pattern: every `useradd`/`groupadd`/`mkdir`/`apt
install` is gated by a "skip-if-exists" check. The `/var/lib/cavecms`
permission bump (750 → 2770 root:cavecmsstate) was specifically
re-runnable across cluster-3's audit cycle.

**Revised scope:** the deploy target is an existing shared server
already running other Next.js apps, not a fresh droplet. The operator
should run **only the cavecms-namespaced blocks** of setup.sh (cavecms user
/ group, dir layout, `/var/lib/cavecms`, age key, pm2-logrotate) and
skip blocks that would conflict with the already-provisioned MariaDB
/ nginx / apt state. The idempotency property still holds for the
blocks that are run.

**Operator action:** dry-read setup.sh's structure, pick the cavecms-only
blocks, run them, then re-run to confirm exit 0 with no new
mutations.

## 2. `install-nginx.sh` and `install-systemd.sh` apply config + reload services without manual edits

**Status:** ✅ verified-in-code · ⚠️ verifiable-only-in-prod (partial)

Both scripts exist and were audited in cluster-1. `install-systemd.sh`
was retroactively extended in cluster-3 commit `06b91c0` to include the
`cavecms-restore-drill` units, and again in `42cff59` to `systemctl start`
(not just `enable`) every timer so they activate immediately rather
than at next reboot.

`install-nginx.sh` at line 128 runs `nginx -t` before reload — config
errors fail the script, never the running nginx.

**Revised scope:** on the shared server, `install-systemd.sh` is
still safe to run (cavecms-namespaced units coexist with whatever else
the box has). `install-nginx.sh` assumes ownership of the nginx
config tree and would conflict with the existing vhosts — the
operator should add cavecms's server block by hand instead. The certbot
deploy hook from `setup.sh:505` still applies and protects against
nginx-config-error reloads.

**Verifiable in prod:** end-to-end test confirms a no-op re-run leaves
nginx + systemd unchanged.

## 3. Certbot deploy hook validates with `nginx -t` before reload

**Status:** ✅ verified-in-code

`scripts/setup.sh:505`:

```bash
nginx -t || { logger -t certbot-deploy "nginx -t failed after renewal; NOT reloading"; exit 1; }
```

The hook is wired into `/etc/letsencrypt/renewal-hooks/deploy/` by
setup.sh and logs to syslog on failure (so the operator sees it via
the `*** SECURITY information ***` cron-mail surface).

## 4. Operator builds artifact locally, ships to server, `scripts/deploy.sh` flips symlink and survives /healthz × 3

**Status:** ✅ verified-in-code · ⚠️ verifiable-only-in-prod

**Revised scope (post-cluster-5):** the GitHub Actions workflows
shipped in cluster-5 commit `b58c3b8` were removed in the post-DoD
revision (operator chose manual deploy via `~/connect/connect` since
the target is an existing shared server, not a fresh droplet). The
DoD item is reframed as **manual artifact build + upload**:

- Local build of the standalone artifact + `pnpm --filter=@cavecms/migrator
  deploy --prod migrator-deploy` to produce a portable migrator
  bundle.
- Tar with zstd at the workspace root as `<sha>.tar.zst`.
- `~/connect/connect <server> upload <sha>.tar.zst` to SCP to the
  server's home dir.
- SSH in, `sudo mv ~/<sha>.tar.zst /opt/cavecms/incoming/`, then
  `sudo /opt/cavecms/current/scripts/deploy.sh <sha> production`.

`scripts/deploy.sh:465-494` polls `http://127.0.0.1:3040/healthz` and
requires 3 consecutive 200s with `CAVECMS_COMMIT` matching the deployed
SHA before flipping the symlink. On failure, it invokes `rollback.sh`
with `AUTO_ROLLBACK=1`.

The manual-build sequence is documented in `README.md` under
"Deploy flow (manual, operator-driven)".

**Verifiable in prod:** first successful manual deploy round-trip.

## 6. DB backup encrypted with age and rotated 30d; restore drill succeeds on a separate machine

**Status:** ✅ verified-in-code · 🟡 operator-action

Cluster-3 shipped `scripts/db-backup.sh` (age-encrypted nightly dump,
local 30-day retention + off-site rclone copy with own retention) and
`scripts/restore-drill.sh` (decrypts last night's dump into a scratch

The on-server weekly drill runs only if `/etc/cavecms/age-identity` is

**always** the canonical recovery-quality gate and operator must run

**Operator action:** decrypt the latest backup on a recovery laptop

## 8. PM2 logrotate keeps last 30d compressed

**Status:** ✅ verified-in-code

`scripts/setup.sh:530-540` installs `pm2-logrotate` and sets:

```
max_size       50M
retain         30          # 30 days
compress       true
rotateInterval 0 0 * * *   # daily at 00:00
```

The values persist across PM2 reloads (stored in `~cavecms/.pm2/module_conf.json`).

## Operator-action summary (gating the production cutover)

The deploy target is an **existing shared server** (not a fresh
droplet) already running other Next.js apps under PM2. The following
must be completed by a human BEFORE flipping DNS at the apex:

1. **cavecms-specific server setup** — run ONLY the parts of
   `scripts/setup.sh` that apply (cavecms user/group, dir layout under
   `/opt/cavecms/`, `/var/lib/cavecms` marker dir, age key block, pm2-logrotate
   block). Skip the apt-install, MariaDB-bootstrap, and nginx-bootstrap
   blocks if those are already provisioned for the other apps.
2. **`/etc/cavecms/env.production`** — populate `DATABASE_URL`,
   `DATABASE_MIGRATOR_URL`, `BACKUP_RCLONE_REMOTE`, reCAPTCHA + SMTP
   keys (mode 640 root:root). `BACKUP_AGE_RECIPIENT` is written by
   `setup.sh`'s age block from `/etc/cavecms/backup.pub`.
3. **Age private key placement** — decide whether `/etc/cavecms/age-identity`
   lives on-server (enables automated weekly restore-drill) or
   off-box (tighter blast-radius; operator runs the drill on their
   own machine).
4. **`~/connect/connect` configuration** — confirm the cavecms target is
   in the server list (`SERVER_IPS`, `SERVER_USERS`, `SERVER_KEYS`).
   Test `~/connect/connect <server>` and `~/connect/connect <server>
   upload <small-file>` round-trip.
5. **nginx vhost** — add cavecms's server block by hand (coexisting with
   the box's existing vhosts). Reload with the certbot deploy hook
   in place (`nginx -t` before reload, per `setup.sh:505`).
6. **Local sanity gates** — `pnpm typecheck && pnpm lint && pnpm test
   && pnpm build && pnpm db:check` all green on the cutover branch.
7. **Initial manual deploy to a staging slug** — build artifact
   locally, upload via `~/connect/connect`, invoke `deploy.sh` on the
   server, verify `/healthz` reports the new SHA. This is the dress
   rehearsal for the cutover.
8. **Off-box restore drill** — confirm clean DB + uploads restore
   from the latest age-encrypted backups on a separate machine.
9. **Smoke-test every public page + every lead-capture form** on the
   staging slug.
10. **Monitoring + alerting** — uptime probe against `/healthz`, log
    shipping if applicable, on-call rotation gets the runbook.
11. **`cavecms-alert@.service` delivery** — confirm it reaches a real
    admin mailbox (postfix/msmtp wired to a deliverable smarthost).
12. **Cutover ritual** — DNS TTL lowered to 60s ≥ 24h before flip →
    Let's Encrypt cert pre-issued via DNS-01 → build + upload + run
    deploy.sh for the production SHA → verify `/healthz` → DNS A
    record flipped → 30 min monitor → TTL restored to 3600s after 7
    stable days.
13. **`git tag v1.0.0`** once production is stable (release marker
    only — no auto-deploy hangs off it).
