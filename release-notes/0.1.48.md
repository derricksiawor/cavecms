## 0.1.48

Stability + security pass on top of 0.1.47. Eight install / admin /
update bugs caught by a codebase-wide debugger sweep, plus three
operator-visible regressions surfaced from the field.

### Safari rendered every install-wizard page unstyled on localhost

`Strict-Transport-Security` + the CSP `upgrade-insecure-requests`
directive were gated only on `NODE_ENV === 'production'`. CLI-installed
sites run with `NODE_ENV=production` but serve over plain HTTP on
`localhost` / `127.0.0.1` / a LAN IP. Safari uniquely honours both
directives on those hosts — the result was every asset upgraded to
`https://`, no TLS listener, full-page unstyled fallback. Chrome
ignored it (per RFC 6761).

Fix: `lib/security/hostKind.ts` classifies the request `Host` into
`loopback` / `lan` / `public`. Middleware now skips both directives on
loopback + LAN even in production, and emits HSTS only on public
hostnames. The HSTS header moved out of `next.config.ts` (where it
couldn't see the request) into middleware. Public-domain installs are
unaffected. Operators upgrading from a prior Safari visit to localhost
will need to clear Safari's HSTS cache once
(`Safari → Settings → Privacy → Manage Website Data → "localhost" → Remove`).

### Operator's custom LOGIN_PATH returned 404

Wizard step 6 saved the operator's chosen hidden-admin path to
`settings.security_login_path`. `getSetting()` is tag-cached for 60 s
via `unstable_cache`. The install helper that wrote the row did not
revalidate the `settings` tag, so every reader — including the
middleware-feeding loopback config endpoint — kept serving the
pre-wizard env default. Operator typed the URL the wizard showed them,
middleware's stale `loginPath` didn't match, the single-segment router
rewrote to `/_page/<custom>`, page lookup returned 404.

Fix: `upsertSetting()` now calls `safeRevalidate(['settings'])` after
the DB write. Same fix applied to every other install / admin /
updates writer that bypassed `upsertSetting()` and built its own
INSERT statement (site-init, complete, template, logo, admin-create,
updates scheduler + notifier). Stale-settings staleness window
collapses from 60 s to "next request."

### "Visit your site" after wizard sent operator back to /install

Middleware caches the security-config snapshot for 3 s in a
globalThis Map the Node-runtime endpoints cannot reach. The wizard's
final POST flipped `install_state.completedAt` but the middleware's
cached snapshot still read `installed: false`. Clicking
"Visit your site" within the 3 s window returned operators to /install
— making it look as if the wizard hadn't completed.

Fix: `/api/install/complete` sets a one-shot `cavecms_just_installed=1`
cookie. Middleware honours it for a single request — force-fresh
fetch of the security-config, then expires the cookie. Race window
closed.

### "Update now" 500'd on legacy installs (`status_path_invalid`)

Sites installed before 0.1.27 don't have `CAVECMS_STATE_DIR` stamped
into `env.production`. The status-file engine fell back to
`/var/lib/cavecms/update-status.json` which the runtime user can't
write to (EACCES) without the `cavecmsstate` supplementary group +
a PM2 daemon re-exec. The apply route mapped the EACCES to a generic
`status_path_invalid`.

Fix: `getInstallStateDir()` now falls back to
`<cwd>/.cavecms-state/` when neither env var is set — derived from
`process.cwd()` which is owned by the runtime user by construction
(otherwise the Node listener couldn't have booted there). The path
allowlist accepts the new prefix, and the apply route explicitly
forwards the resolved path to the orchestrator script via
`CAVECMS_UPDATE_STATUS_PATH`. The shell script's allowlist was
extended to match. The apply route also now distinguishes EACCES
(`state_dir_not_writable` — operator-actionable) from path-allowlist
failures (`status_path_invalid` — env misconfig).

### Cookie SameSite — `strict` → `lax`

Session, CSRF, and JTI cookies all defaulted to `sameSite: 'strict'`.
That blocks the cookie on cross-site top-level GET navigations,
which silently breaks every OAuth callback (Slack / Google / GitHub /
Stripe Connect) before the operator can integrate one. CSRF
protection comes from the dedicated double-submit token, not from
SameSite — so loosening to `lax` doesn't weaken defence. Forward-
compatibility fix.

### Admin leads + newsletter tables broke on Cloudflare email
obfuscation

Public `SiteFooter` was migrated to `<CfSafeMailto>` in 0.1.12. The
admin Leads + Newsletter row drawers still emitted raw
`<a href="mailto:…">`. Operators with Cloudflare Email Address
Obfuscation enabled for the zone hit a React #418 hydration error
and a CSP-blocked decode script on every drawer open. Swapped to
`<CfSafeMailto>`.

### `app/%5Fpage/[slug]/page.tsx` was nearly renamed to a dead route

A mid-session "fix" renamed the URL-encoded `%5Fpage` directory to
`_page`. Next.js treats `_<folder>` as a private folder (non-routable
per official docs), so the rename would have made every CMS page
404. Caught + reverted by the debugger pass. Mentioned here because
the rename rationale ("the encoding looks wrong") will tempt the same
mistake again — the encoding is the documented Next.js convention.

### Operator action

Anyone running 0.1.45–0.1.47 on a Safari that already cached HSTS
for localhost: clear it once via
`Safari → Settings → Privacy → Manage Website Data → "localhost" → Remove`.
Chrome + Firefox are unaffected.

### CLI

`create-cavecms@0.1.48` published in lockstep, no functional change
from 0.1.47.
