## 0.1.3

Second fix-up of the 0.1.1 release pipeline. 0.1.2 added `mysql2` +
`drizzle-orm` as `serverExternalPackages` so they'd ship as discrete
node_modules entries — but the build-zip's `dereference: true`
collapsed pnpm's relative-symlink layout into top-level real
directories WITHOUT carrying the sibling-dep chain pnpm parks
alongside each package in `.pnpm/<hash>/node_modules/`. So `mysql2/`
was on disk in the install but `require('sql-escaper')` (a transitive
dep) failed because Node's resolver had no way to find it.

### Fix

`build-zip.mjs` now uses `cp -a` for the `.next/standalone` entry
(instead of Node's `fs.cpSync`). `cp -a` preserves relative symlinks
verbatim, so `node_modules/mysql2 → .pnpm/mysql2@.../node_modules/mysql2`
stays put as a relative link and the customer's `unzip` restores it
correctly. Combined with the (separately-shipped) `.pnpm/` content
store, Node's resolver follows the symlink chain to find every
transitive dep at runtime.

### Why `cp -a`, not `dereference: false` in cpSync

Tried that first. Node's `fs.cpSync` with `dereference: false`
absolutifies the symlink target on copy (turning
`node_modules/mysql2 → .pnpm/...` into `mysql2 → /Users/<dev>/...`)
which then breaks on the customer's box where that absolute path
doesn't exist. `cp -a` is the only mode that keeps the relative
target verbatim.

### Operator impact

- Fresh installs via `npx create-cavecms` now reach step 6
  (migrations) AND step 7 (PM2 start) cleanly.
- In-app updater applies cleanly on top of 0.1.1 / 0.1.2 — no schema
  change. The in-app update path uses the same release zip, so 0.1.2
  installs that need to update would benefit from re-installing via
  the CLI to get the corrected node_modules layout, OR the next in-app
  update will land 0.1.3's zip atomically and the runtime symlinks
  reshape.
- Runtime behaviour unchanged.

### Nothing else changed

Same scope as 0.1.2 — release-pipeline correctness only.
