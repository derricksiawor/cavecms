## 0.1.42

Fixes the React 19 + Server Components hydration crash on template
pages that ship multiple slide-up animated widgets.

### Fixed

**Non-home template pages (/dining, /story, etc.) 500'd with
"React.Children.only expected to receive a single React element
child"** even though every widget renderer passes a single root
element to `MotionTarget`. Root cause is the React 19 + Next.js 15
Server Components hydration quirk that the Radix UI team also hit
in `@radix-ui/react-slot` (their v1.2.0 patch). The
`cloneElement(Children.only(children), { ref })` pattern is "soft
deprecated" per Sebastian Markbåge of the React core team —
"cloneElement is basically soft deprecated. It works against any
ability to optimize by inlining." Concretely, `Children.only`'s
validator throws unpredictably during hydration when the upstream
React reconciler has already unwrapped a server-stream fragment
past its inspection point. The error fires depending on the
ordering of sibling MotionTarget instances on the page — which
explained why some hotel-solenne sub-pages crashed and others
rendered fine despite identical widget shapes.

Replaced the validator with `isValidElement` + `cloneElement` so
the ref attaches whenever the child is a real DOM-backed element,
and silently degrades to "no animation" when it isn't (instead of
crashing the page render). Every existing call site keeps working
without changes because every caller already passes a single
element — the validator was the only thing causing failures.

### Migration impact

None — runtime behavior change only.
