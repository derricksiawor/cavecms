## 0.1.51

The real fix for the CMS-slug 404 — plus generic legal pages on every
install.

### Every non-reserved CMS slug 404'd (the actual root cause)

0.1.49 renamed the internal render route `/_page` → `/cms-render`
believing the underscore-prefix was the cause. It wasn't, and the
404 persisted. The true mechanism, confirmed by patching the live
standalone:

In Next.js 15.5 **standalone**, `NextResponse.rewrite()` to a path that
the middleware matcher also matches causes middleware to **re-execute
on the rewritten path**. So when a visitor hit `/agents`, middleware
rewrote it to `/cms-render/agents` — and then ran a SECOND time against
`/cms-render/agents`. The `refuseInternalPageRoute` guard (whose job is
to hide the internal prefix from external probing) fired on the app's
OWN internal rewrite and returned an empty-body 404. Every CMS page
died this way; Safari content-sniffed the bodyless 404 as a binary and
downloaded it.

Reserved slugs (`/`, `/contact`, `/blog`, `/privacy`, `/projects`)
worked because they resolve to real filesystem routes and never go
through the rewrite. That's why only home + contact rendered.

**Fix:** the guard now distinguishes the internal rewrite re-entry from
a genuine external hit via a marker request header (`x-cavecms-render`)
that only our own rewrite sets (and which we strip from inbound
requests to prevent spoofing). External hits to `/cms-render/*` still
404; the internal re-entry passes through and renders. Verified on the
live test install: `/agents` `/listings` `/neighborhoods` `/contact`
all 200, direct `/cms-render/agents` still 404.

### Every install shipped blank Privacy + Terms pages

The migration creates the `privacy` + `terms` page rows (preserved
across template picks as legal pages), but ships **zero content
blocks**. The generic legal copy only seeded via the contributor
`db:seed` path — a customer CLI install never ran it, so both pages
were blank. A footer "Privacy" link led to an empty page.

**Fix:** the generic, brand-neutral legal content (Privacy Policy +
Terms of Service, with `[Your Company]` placeholders the operator
edits) moved to a shared pure-data module
(`lib/cms/siteTemplates/legalContent.ts`). The install-template step —
which every install runs, including Skip/default — now seeds that
content into the legal pages when they're empty. Non-destructive: a
page that already has blocks (operator-edited, or a re-pick) is left
untouched. Applies to all 8 templates uniformly since legal pages are
shared, not per-template.

Existing installs upgrade via Settings → Updates → Update now.
