## 0.1.47

Architectural fix on top of 0.1.46 — the release pipeline no longer
touches the network at build time. Time-critical security patches
ship regardless of Unsplash availability.

### The problem

0.1.45 through 0.1.46 deploys re-downloaded all 102 template-photo
originals from `images.unsplash.com` on every release build. Three
0.1.45 deploys failed for network reasons:

1. Sharp's 24 MP input-pixel cap tripped on a 4000×6001 portrait crop
   (architecture: bounded for upload-bomb defence, not adjusted for
   the curated release-build path).
2. Unsplash's CDN dribbled bytes at 200–400 KiB/s on the publisher
   box; 30 s per-image fetch budget timed out.
3. Bumping to 90 s + 50 MP cap let it through, but the architecture
   was wrong from the start — a flaky third-party CDN had a hard
   gate on customer-facing releases.

The blast radius: **a time-critical security patch (XSS, RCE,
auth-bypass) blocked behind Unsplash availability**. Unacceptable on
a CMS.

### The fix

`template-media-cache/<slug>/<key>.<ext>` — the canonical originals
now live committed alongside source code. Same trust model as
`/public/icons/social/*.svg`: vendored, deterministic,
offline-buildable.

- **Release pipeline (`pnpm release:build-template-media`)** READS
  from the cache and runs sharp. **Zero network. Zero fetch. Zero
  retry budget.** If a cache entry is missing, the build dies
  loudly with `"<key> not in template-media-cache — operator must
  run pnpm template-media:hydrate"`. There is no silent fallback.
- **Operator workflow (`pnpm template-media:hydrate`)** downloads
  Unsplash originals into the cache. Run ONCE when editing a
  `media-sources.json` entry, commit the new cache file, deploy.
  The hydrate script is the only file in the entire system that
  touches images.unsplash.com.
- **`scripts/release/build-template-media.ts`** lost its
  `httpsGetBuffer` helper and `downloadOriginal` function entirely
  — there is no longer a path through the build script that could
  even attempt a network call.

### Operator implications

- **First-time clone of the repo**: ~240 MB heavier (102 photos at
  ~2.4 MB avg). One-time cost.
- **Subsequent deploys**: media-build phase drops from ~7 minutes
  (downloading) to ~1 minute (sharp only). Network is no longer in
  the critical path.
- **Adding a new template photo**: edit `media-sources.json` → run
  `pnpm template-media:hydrate --slug=<slug> --key=<key>` → commit
  the new cache file → deploy.
- **Refreshing all photos**: `pnpm template-media:hydrate --force`.

### Migration impact

None — variants the release zip ships are byte-identical to 0.1.46
(same sharp pipeline, same input bytes). Customers see no change.

### CLI

`create-cavecms@0.1.47` published in lockstep, no functional change
from 0.1.46.
