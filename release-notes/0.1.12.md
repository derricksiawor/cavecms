## 0.1.12

Closes the two console errors documented as "deferred / known
limitations" in 0.1.10: the Cloudflare email-decode CSP violation
and the React #418 hydration text mismatch. They turned out to be
the same root cause — Cloudflare's Email Address Obfuscation
rewriting the SSR HTML before it reaches the browser — and a single
fix closes both.

### Fixed

**Public-site contact email rendering triggers Cloudflare Email
Address Obfuscation.** When the site sits behind Cloudflare with
Scrape Shield → Email Address Obfuscation on (the zone default),
Cloudflare's edge rewrites every `<a href="mailto:…">…@…</a>` in the
response to a `/cdn-cgi/l/email-protection#…` link plus the literal
text `[email protected]`, and injects
`/cdn-cgi/scripts/<build>/cloudflare-static/email-decode.min.js` to
decode it on the client. Two downstream failures fell out of that:

1. **CSP violation (formerly bug #27).** The injected script
   carries no nonce, so our `script-src 'nonce-…' 'strict-dynamic'`
   policy refuses to run it. Visitors saw `[email protected]`
   gibberish where the email should be, and a noisy CSP error in the
   browser console.
2. **React hydration mismatch (formerly bug #28).** The rewritten
   SSR HTML carries `[email protected]` text where React's
   in-memory tree still holds the real email string. On hydration
   React detects the text-node mismatch and throws minified error
   #418 — the affected subtree gets discarded and re-rendered on the
   client. Functional but noisy in the console.

Both symptoms disappear if Cloudflare can't see a `mailto:` href or
a plain-text email pattern in the SSR HTML. 0.1.12 introduces a
small client component `components/CfSafeMailto.tsx` that:

* Server-renders an empty `<span>` (no `mailto:`, no email text →
  Cloudflare's pattern matchers find nothing to rewrite, no script
  injection).
* Flips to the real `<a href="mailto:…">…@…</a>` on the client after
  the first `useEffect`. Cloudflare can't reach client-side renders.
* Uses `useState(false)` for the initial state so the SSR and the
  client's first render produce identical HTML — no hydration
  warning on the placeholder itself.

`components/SiteFooter.tsx` now uses `<CfSafeMailto>` for the contact
email line. Tradeoff: no-JS visitors see the empty placeholder
instead of the email. Cloudflare's own EAO behaves the same way
(replaces the email with `[email protected]` text and decodes via
JS), so we're not regressing the no-JS case — and we keep the CSP
intact.

### Out of scope this release

`components/blocks/LxChannelCard/render.tsx` still renders `<a href={mailto:…}>`
when an operator uses a channel card for an email. The default
one-pager seed doesn't include any channel cards, so the immediate
customer-journey repro is closed; a structural revisit for the
channel-card block ships in a later release.

### Migration impact

None. Single new component + a one-line replacement in the public
footer. Existing data, settings, and migrations are untouched.
