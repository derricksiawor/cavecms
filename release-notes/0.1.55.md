## 0.1.55

Editor + auth fixes: blocks stay legible on any section, widgets always
land in a proper section/column, the page editor's quick-add only offers
real blocks, and a locked-out login now tells you how long to wait.

### Blocks read correctly on any section background

Two blocks rendered light-on-light (or dark-on-dark) when their tone didn't
match the section they sat on — most visibly the **Featured Projects** grid,
whose project names + taglines were near-invisible on a light section.

- **Featured Projects** now auto-contrasts its section: light text on dark
  sections, dark text on light ones. There's no per-block tone to mismatch
  the background anymore — drop it on any section and it stays legible.
- **Icon box** (`champagne-outline`) is now section-aware too: a clean
  transparent card with a champagne hairline border on dark sections (it
  used to be a translucent ivory fill that decayed to muddy gray on dark).

### Adding a widget always builds a proper section → column → widget

Clicking a widget on an empty page used to create a loose top-level widget
that then failed to save. Now every widget insert guarantees the correct
structure — when there's no column to drop into, a section + column are
created and the widget nests inside. If that setup can't be created, the
insert is cancelled with a clear message instead of leaving a broken block.

### Page editor "Add block" only offers real blocks

The admin page editor's quick-add menu still listed removed legacy block
types (text, cta, quote, …) that would have created blocks with no renderer.
It now offers current editorial primitives (Heading, Text, Action, Quote),
sourced from the same seed data the schema validates — the full block
library remains in the on-page inline editor.

### Login tells you when you're locked out

A rate-limited or temporarily-locked-out sign-in showed the same generic
"Invalid email or password" as a wrong password, leaving operators stuck
with no idea why. It now returns a clear "Too many failed attempts — try
again in about X minutes" with a `Retry-After`, while a genuine wrong
password keeps the generic message. The lockout message is enumeration-safe:
it's driven by the submitted email/IP regardless of whether the account
exists, so it never confirms an account.
