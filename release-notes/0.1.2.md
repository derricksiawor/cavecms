## 0.1.2

Fixes the `mysql2 not found` error every fresh `npx create-cavecms`
install hit at the migrations step.

### What was broken

`scripts/install-migrate.mjs` is a standalone Node script the CLI runs
right after writing `env.production` to apply the bundled Drizzle
migrations against the operator's MariaDB. It uses `createRequire` to
resolve `mysql2/promise` from the unpacked release zip's
`node_modules/`. But 0.1.1's zip didn't actually have `mysql2` or
`drizzle-orm` as discrete packages on disk — Next.js's standalone
build had inlined them into webpack chunks instead. The server.js
runtime could still query the DB (via the bundled chunks), but the
out-of-band install-migrate script had nowhere to load the driver
from, and the install died with:

```
[install-migrate] mysql2 not found — looked in:
  …/.next/standalone/package.json
  …/scripts/install-migrate.mjs
```

### Fix

`next.config.ts` now marks `mysql2` + `drizzle-orm` as
`serverExternalPackages`. Next ships them as physical packages under
`.next/standalone/node_modules/` (next to `sharp`, `nodemailer`, etc.)
instead of inlining their code into webpack chunks. The release zip
carries the directories, `install-migrate.mjs`'s `createRequire`
resolves them, and migrations now run as designed.

### Operator impact

- Fresh installs via `npx create-cavecms` no longer fail at step 6.
- Existing installs are unaffected — their bundled-chunks runtime
  works the same as before. The in-app updater applies 0.1.2 on top
  of 0.1.1 cleanly (no schema change in this release).
- Bundle size: marginally smaller, since `mysql2` + `drizzle-orm` are
  no longer duplicated across webpack output AND `node_modules/`.

### Nothing else changed

This is a pure release-pipeline fix. No new features, no behaviour
changes in the running app.
