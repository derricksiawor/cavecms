## 0.1.31

In-app updater hotfix — extract step.

### Fixed

**Tarball extract refused on missing `pnpm-lock.yaml`.** The
orchestrator's `extract_tarball_atomic` (line 400) checked for
`pnpm-lock.yaml` in the extracted release and returned error code 3
when missing. CLI release zips never shipped one (it's a source-tree
file; the released artifact carries vendored `node_modules` inside
`.next/standalone/`), so every in-app update on a CLI install failed
at step 2 with "missing package.json or pnpm-lock.yaml". Updated the
check to require only `package.json` + `.next/standalone/server.js`
— both of which CLI releases DO ship.

Also expanded the move-aside path list to include `.next` so the new
tarball's build fully replaces the old one. Previously the old
`.next/standalone/` overlayed with the new chunks, mixing stale and
fresh build artifacts.

**Release zips now include `pnpm-lock.yaml`.** Strictly redundant for
CLI installs (vendored), but older orchestrators (every install
between 0.1.16 and 0.1.30) check for it in the extracted tarball.
Shipping the lockfile (~275 KB) makes existing in-the-wild installs
compatible with every future release without requiring a fresh
reinstall.

### Migration impact

Customers on 0.1.16-0.1.30 CLI installs can now apply 0.1.31+ via
the dashboard's in-app updater — the lockfile-presence check that
was blocking them is satisfied by the new zip layout. 0.1.31+
orchestrators no longer require the lockfile at all.
