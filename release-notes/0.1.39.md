## 0.1.39

A thorough review pass on 0.1.38 surfaced a latent schema bug that
zero-blocks every install + several install-route hardening issues
in the new template-media seam. This release ships the migration +
all CRITICAL / HIGH / MEDIUM / LOW fixes.

### Fixed

**Pre-existing schema bug: `deleted_at` defaults silently soft-delete
every row.** Migrations 0002, 0004, 0007, 0008 declared `deleted_at`
columns as `timestamp(3)` with no explicit `NULL` / `DEFAULT`. Under
MariaDB's legacy `explicit_defaults_for_timestamp=0` the FIRST
TIMESTAMP column in such a table is silently promoted to `NOT NULL
DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3)`, so
every INSERT lands a row that's already soft-deleted. Affected tables:
`content_blocks`, `media`, `leads`, `posts`, `projects`. Migration
0023 fixes the column shape (`NULL DEFAULT NULL`) AND repairs every
existing soft-deleted row (`UPDATE … SET deleted_at = NULL`). Latent
since migration 0002 — 0.1.38 is the first release where the install
template route exercised a non-trivial seed end-to-end and made the
bug operator-visible (public home page rendered the "Launching soon"
splash because hydratePage's `deleted_at IS NULL` filter returned
zero blocks). `pages.deleted_at` was already correct (fixed by
migration 0010).

**Install route mutated the SITE_TEMPLATES registry singletons in
place.** `injectMediaRefs` was setting `widget.data[field]` and
deleting `widget._imageKeys` on the same objects returned by
`getTemplate()`. After the first install, every subsequent re-pick in
the same Node process saw widgets with `_imageKeys` already stripped;
manifest pre-flight returned "no media keys" so `provisionTemplateMedia`
ran no-op, and widgets were inserted carrying baked-in stale media_ids
from the previous install. Fixed by `structuredClone(template)` at the
top of the POST handler. The site-template registry is now also
deep-frozen at module load — a future contributor who forgets the
clone gets a loud TypeError at the mutation site instead of a quiet
data-corruption bug.

**Re-pick leaked previous template's media rows + variant files.**
The wipe-and-reseed dropped pages + content_blocks for the previous
template but never cleaned the bundled `media` rows or their
`media_references`. After an operator re-picked, the Media Library
carried unreachable "ghost" entries that returned 409 from DELETE
(pinned by orphan media_references pointing at vanished content_blocks).
Now: a `reapPreviousTemplateMedia` step runs inside the wipe TX,
identifies bundled rows via `original_name LIKE 'template/%'`,
cascade-deletes them, and unlinks the variant files AFTER the TX
commits (rollback-safe).

**Variant filename + path-containment validation at the parse
boundary.** The bundled manifest's `variants.{thumb,md,lg,og}` and
`key` fields are now constrained to a strict regex; `path.resolve` is
asserted to stay inside the per-template bundled-media root before
every `copyFile`. The signed-zip envelope upstream eliminates the
realistic exploit today; this is defense-in-depth at the parse gate
per project Security Standards.

**Manifest `templateSlug` cross-check.** A misplaced or swapped
manifest file would have silently installed one template's imagery
under another's pages. The install route now refuses with 500
`template_media_manifest_slug_mismatch` if the bundled manifest's
`templateSlug` doesn't equal the operator-selected slug.

**Width / height / byteSize bounds at the Zod gate.** Defends against
a future manifest carrying a value larger than the `int` column's
2,147,483,647 ceiling (would either trip a `Range error` mid-TX or
silently truncate). Caps: dimension 24,000 px (matches sharp's
`LIMIT_INPUT_PIXELS`); byteSize 50 MiB.

**Variant files now get explicit `chmod 0o640` after copy.** Without
this the file mode depended on the cavecms-user umask. A customer
install with a hardened umask (077) would have landed files at 0600 →
nginx 403 → broken template imagery on first paint.

**Variant rollback cleanup wrapped in try/catch.** A synchronous
throw in the cleanup loop (path-resolution edge case) can no longer
mask the original transaction error in operator-facing logs.

**`media_references` derived from `collectMediaPaths(cleaned)`** —
same canonical extractor used by saveBlock + hydratePage. Closes a
contract-drift risk where a future nested-MediaRef block (e.g.
`gallery[3].image`) could have rendered the install route's reverse-
index incompatible with the rest of the system. Switched from `ON
DUPLICATE KEY UPDATE media_id = media_id` to `INSERT IGNORE` to match
the blocks route's pattern.

**Widget meta now validated at install time via `WidgetMetaSchema`.**
A typo in a template's `meta: { marginTop: 'smm' }` would have landed
in DB and crashed the renderer. SectionMeta + ColumnMeta were already
validated; WidgetMeta closes the symmetry.

**`preservedLogo` validated via Zod before re-persistence.** A hand-
edited DB cell with a malformed `site_header.logo` shape no longer
gets re-persisted verbatim. Parse-failure falls back to "no preserved
logo" (the wordmark renders).

**Release-builder `httpsGetBuffer` gained a wall-clock timer + an
`aborted` listener.** `req.setTimeout` only catches socket inactivity;
a slow-loris origin could have held the connection open dribbling one
byte per timeout window. Now bounded by a wall-clock that destroys
the request regardless of how the bytes are paced.

**Sharp pipeline timeout parameterized.** Upload route keeps the 10s
budget (user is waiting). The release-time bundler gets 30s — slow
laptop, cold libvips, network hiccups shouldn't brick a release.

### Added

**Router-cache invalidation after install/template commits.**
`revalidatePath('/', 'layout')` busts the Next.js router cache for
the full route tree so newly-seeded pages don't land on stale 404
frames during soft-nav lookups.

**Hôtel Solenne reservations page now ships a cover photo.** Vintage
brass service bell on a wooden front desk (Jonathan Cosens
Photography, Unsplash). The page operator-conversion lands on — visual
completeness matters most there.

**Wizard error copy for `template_media_*` codes.** Operator now sees
"This template's images couldn't load on your install — pick a
different template, or Skip to install with no template." instead of
the generic "try again" fallback. Same treatment for the rarer
`preserved_legal_pages_missing`.

### Migration impact

Migration 0023 runs on every install / update boot. Repair UPDATE
affects only rows whose `deleted_at` was set by the buggy default —
real soft-deletes (from the soft-delete admin path) leave behind a
deleted_at that NEVER matches a `created_at` from a system-page seed,
but to be safe the migration UPDATEs unconditionally: every existing
soft-deleted row gets brought back to live. This is intentional —
the bug masked an entire DB worth of "deleted" rows that were never
actually deleted, and customers running 0.1.38 or earlier should see
their pages + media restored to life on upgrade. Operators who DID
manually soft-delete content via the admin UI will need to re-delete
those specific rows post-upgrade (rare path; documented as a known
upgrade behaviour rather than a bug).
