## 0.1.86

Build **dropdown menus** in your header with a new drag-and-drop menu builder, and agent editing gets dramatically faster — your AI tool can now make a whole page's worth of changes in a single request, plus the API can read everything it edits.

### New — dropdown menus in your header

Your top navigation can now have **dropdowns**. Under **Settings → Header**, the navigation editor is now a drag-and-drop **menu builder**: add a link, then add sub-links beneath it — reorder by dragging, or use the nest button to tuck one link under another. Any link that has sub-links automatically becomes a **dropdown** on your live site — a hover menu on desktop, a tap-to-expand section in the mobile menu — with nothing to wire up: the dropdown takes its light/dark look from your **Header tone** and its colours from your **Theme palette**, so it always matches the bar. Leave a link's address blank to turn it into a pure dropdown trigger that just opens its sub-links. (Up to 6 top-level links and 12 sub-links each, so the bar stays uncluttered.)

The same builder now powers your **footer columns** too — add a column heading, then its links, with the same drag-to-reorder and promote controls.

For headless front-ends and automation there's also a new **menu API**: `GET /api/cms/nav` returns your header and footer menus as JSON (public, no key needed), and an authenticated `PUT` replaces a menu in a single call.

### Faster — edit a whole page in one request

When an AI agent edited your site through the API, it had to send one change at a time, in strict order, waiting for each before the next. A dozen small edits meant a dozen round-trips. Now there's a **batch endpoint**: your agent sends many edits — recolor sections, rewrite headings, add new blocks, delete others, reorder a column — in **one request that applies as a single, all-or-nothing change**. "Repaint twelve sections and rewrite five headlines" goes from seventeen calls to one. It's as fast as editing the database directly, but every change is still fully validated, sanitized, and written to your audit log — so speed costs you nothing in safety. If anything in the batch is wrong, the whole thing is rejected and nothing changes.

Two touches make it especially quick for agents: a change can **merge a single field** (set a section's background without re-sending its column count and spacing), and it can **build a whole section → column → widget tree in one go**.

### The API can now read everything it can change

A couple of gaps meant an agent could change things it couldn't first read back — a section's background, a blog post's body, a project's details. Those are closed: reading a page now returns each section's full styling, and there are direct ways to read a single post or project before editing it. Your agent can now reliably read, change, and confirm any piece of content.

These are improvements for AI-agent and API editing — the visual builder and everything else work exactly as before.

### New & updated docs

The **AGENTS.md** guide in your install — the file Cursor, Claude Code, Windsurf, and Codex read automatically — now documents the batch endpoint and the read-before-edit routes, so your agent reaches for the fast path on its own. Online, there's a complete new **API reference** (authentication, the batch endpoint, every block type, an error table, recipes) plus a step-by-step **deploy guide** for taking a site from a local build to a production server.

### Also fixed

- A duplicate value that should be caught (a repeated page slug, a re-used anchor id, an email already in use) now returns a clear, specific error instead of a generic server error — so tools and forms can react to it cleanly.

Updating applies everything automatically.
