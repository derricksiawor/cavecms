## 0.1.1

A reliability-focused patch release. Hardens the in-app updater against
mid-flight failures, surface drift, and post-completion crashes, and
strengthens the release-integrity pipeline so every published zip can be
verified offline against a bundled Ed25519 public key.

### In-app updater

- **Snapshot-based rollback** uniformly across git + tarball modes. Every
  apply takes a filesystem snapshot of the install dir + a mysqldump of
  the live DB at step 2 (before any change touches disk). A failure at
  any later step restores both atomically, so a half-applied update
  never leaves the operator's site in a wedged state.
- **Post-completion watchdog**. At the end of every successful apply a
  detached `cavecms-watchdog.sh` polls `/healthz?verbose=1` for one hour.
  Three consecutive failures (or a commit-SHA mismatch) acquire the
  update lock and roll the install back to its pre-update snapshot — so
  delayed crashes (env-shape drift, lazy-loaded import explosion, slow
  DB migration that times out at first traffic) don't strand the
  operator at 3am.
- **Memory pre-flight**. `step_preflight` now refuses to start the
  apply if `MemAvailable` is under 512 MB. A `pnpm build` on a 256 MB
  VPS would otherwise OOM mid-step and leave the install half-rebuilt
  with no clean retry path.
- **Stronger detach mechanism**. The orchestrator now picks the
  strongest available "outlive the parent" primitive in this order:
  `systemd-run --user --scope` → `setsid + nohup` → bare `nohup`. The
  chosen mechanism is logged into the audit row + surfaced to the
  watchdog via env so a child can never be killed by an SSH disconnect
  mid-update.
- **Lock-staleness detection** now cross-checks `/proc/<pid>/cmdline`,
  defeating PID recycling and cross-user EPERM-stall edge cases.

### Release-integrity pipeline

- **Ed25519 signing** is wired end-to-end. `pnpm release:build` signs
  the zip + manifest with `~/.cavecms-release-private.pem`; the public
  key is bundled directly into the `create-cavecms` CLI (no
  round-trip to fetch it) AND served at
  `https://cavecms.derricksiawor.com/pubkey.pem` for rotation
  contingencies. A tampered or replayed zip now fails verification
  with a refuse-to-install error.
- **`CAVECMS_COMMIT` sealed into env.production** at install time so
  the in-app updater can compare "running version" against "latest
  release manifest" out of the box. Pre-0.1.1 installs reported
  `sha='dev'` and the updater refused with `cannot_apply_from_dev`.
- **Per-version release notes** in `release-notes/<version>.md`. The
  build script reads this file directly into the manifest's `notes`
  field, so operators see real changelog copy in the Updates modal
  instead of the placeholder `"Release notes — fill in before
  publishing."` text that earlier releases shipped with.
- **Zod-validated manifest fetch**. `checkLatestRelease` now validates
  every field of the `/updates/latest.json` payload before trusting
  it, defending against a tampered CDN edge or operator-supplied
  release host that serves a malformed body.

### Install wizard

- Middleware bypasses `/icons/*.svg` and other static assets during
  the pre-install gate (the Wordmark mark was rendering as a broken
  image on the first wizard step).
- Wordmark tone switches to `dark` on the cream-bg wizard surface so
  the mark reads correctly.
- `suppressHydrationWarning` on `<html>` to silence browser-extension
  attribute injection warnings.
- The "use install token" banner now only shows when the server-side
  gate actually requires the token (contributor-dev sees no banner).

### Operational

- `setup.sh` provisions `/var/lib/cavecms/snapshots/` at install time
  and installs `deploy/logrotate.d/cavecms` into `/etc/logrotate.d/` —
  the ~20 distinct log streams from the orchestrator + watchdog no
  longer accumulate ~50 MB/year on a weekly-update install.
- `cavecms-update.sh` uses `df -kP` for POSIX-portable disk checks and
  a `python3` `WATCHDOG_UNTIL` calculation that works on hosts without
  GNU or BSD `date`.
- A `CAVECMS_MAINT_TOGGLED_BY_US` sentinel prevents the orchestrator's
  trap handlers from clobbering operator-set maintenance mode.
