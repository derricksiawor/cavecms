## 0.1.16

Two bugs that combined to make the in-app updater unusable on
CLI-installed sites — fixed.

### Fixed

**In-app updater 500'd with `status_path_invalid` on fresh CLI
installs.** The status file + cross-process lock the apply route
acquires lives at `/var/lib/cavecms/update-status.json` by default.
That directory is mode 2770, group `cavecmsstate`, and the CLI
installer adds the runtime user (e.g. `www-data`) to that group at
install time. But Linux loads supplementary groups when a process
starts — the already-running PM2 daemon doesn't pick up the new
group until it re-execs, and `pm2 restart <app>` doesn't help
because the daemon is the parent. The CLI did warn about this but
operators shouldn't need a daemon-wide restart for in-app updates
to work at all.

Fixed by introducing a per-install state directory
(`CAVECMS_STATE_DIR`, written into `env.production` by the CLI and
provisioned at `<install dir>/.cavecms-state/`). The runtime user
owns it directly — no supplementary-group dance required. The
status-path allowlist in `lib/updates/statusFile.ts`,
`scripts/cavecms-update.sh`, and `scripts/cavecms-watchdog.sh` all
accept paths under this dir. Existing bare-metal deploys keep
working — `/var/lib/cavecms/` remains in the allowlist and is
preferred when `CAVECMS_STATE_DIR` is not set.

**Release notes shipped placeholder text.** The build script's
manifest writer was hardcoding `## <version>\n\nRelease notes —
fill in before publishing.` into the `notes` field. That's the
text customers saw in `/admin/settings/updates` as "What's new" —
useless. The build now reads `release-notes/<version>.md` from
disk and embeds its contents into the manifest. A missing file
falls back to the placeholder with a warning at build time, so
the publisher gets a visible reminder.

### Migration impact

Customers re-installing from CLI 0.1.7+ get the new `CAVECMS_STATE_DIR`
env var automatically. Existing installs (< 0.1.16) keep working
against `/var/lib/cavecms/` — if their PM2 daemon was restarted
after the cavecmsstate group was added, in-app updates work; if
not, they hit the bug this release fixes. Recommended path for
existing affected installs: re-install from 0.1.16+.
