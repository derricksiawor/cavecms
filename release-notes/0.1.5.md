## 0.1.5

Fixes the customer install's migration step on releases that have
migration `.sql` files outpacing the Drizzle journal.

### What was broken

`db/migrations/` had 21 `.sql` files but
`db/migrations/meta/_journal.json` only listed the first 11 (0000-0010).
10 newer migrations (0011-0022 with gaps at 0018, 0020) had been added
as raw `.sql` files without re-running `drizzle-kit generate`. The dev
migrator wrapper (`pnpm db:migrate` → `drizzle-kit migrate`) tolerated
the gap on existing dev DBs (already had everything applied + tracked
by hash), but `install-migrate.mjs` (the customer-install migrator) is
strict — it refused with `The release zip is inconsistent` and
`process.exit(70)` rather than risk a partial schema.

### Fix

Backfilled the journal entries for the 10 orphan migrations. Each gets:

- `idx`: parsed from the filename prefix (0011 → 11, …, 0022 → 22)
- `tag`: filename without `.sql`
- `version`: "5" matching every existing entry
- `when`: file mtime in milliseconds
- `breakpoints`: true

install-migrate iterates entries by ascending `idx`, applies each in
order. The gaps at idx 18 + 20 are real (no matching `.sql` files
exist) and harmless — the iterator just skips them.

### Operator impact

- Fresh `npx create-cavecms` installs now complete through migrations.
- Existing dev DBs are unaffected — `db-migrate-with-lock.ts` was
  already tolerant of the orphan files, and the `__drizzle_migrations`
  table tracks by content hash, not journal idx. Re-running
  `pnpm db:migrate` on a dev DB after this change is a no-op (every
  migration's hash is already recorded).

### Follow-up worth doing later (not this release)

The `db:migrate` workflow should fail loudly if `_journal.json` is
out of sync with the `.sql` files on disk, OR `db:generate` should be
run automatically when new migrations land. The current gap was
silent — only the customer-install path caught it.
