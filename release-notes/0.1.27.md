## 0.1.27

In-app updater on CLI-installed sites — architectural fix.

After 0.1.16, 0.1.18, 0.1.24, and the (unreleased) 0.1.26 each closed
one symptom of the updater stalling on CLI installs, an end-to-end
audit surfaced the real shape of the problem: the orchestrator was
written for the bare-metal `deploy.sh` flow (git pull + `pnpm install`
+ `pnpm db:migrate` + `pnpm build`), but CLI installs ship as
pre-built standalone tarballs with vendored `node_modules` inside
`.next/standalone/`, no `pnpm-lock.yaml`, no `pnpm` binary on the host,
and no git history. Every "fix" so far was patching one symptom of
that mismatch; the next step always tripped over the next
incompatibility. This release fixes the root cause once.

### Fixed

**Orchestrator now branches on tarball-vs-git mode at every destructive
step.** `scripts/cavecms-update.sh` was a single linear flow assuming
the bare-metal layout. It now branches on `CAVECMS_UPDATE_TARBALL_URL`:

* **Step 3 install:** tarball mode skips `pnpm install --frozen-lockfile`
  entirely — the standalone bundle's `node_modules/` came with the
  extracted tarball.
* **Step 3 migrate:** tarball mode runs `node --env-file=$ENV_FILE
  scripts/install-migrate.mjs` (mysql2 + journal-walking, identical
  state-tracking to Drizzle CLI). The shipped CLI uses the same
  script at fresh-install time, so the migration state row format
  is unchanged.
* **Step 4 build:** tarball mode skips `pnpm build` — the tarball IS
  the built artifact. The static + public symlinks are still re-linked.
* **Rollback** mirrors the same branch: tarball mode skips the
  `pnpm install --frozen-lockfile` re-hydration, since the snapshot
  already captured `.next/standalone/node_modules/`.

Net effect: customers on the PM2 / systemd / manual surfaces never
need pnpm, never need a build toolchain, never need git. The
orchestrator only invokes what their host actually has.

**Per-install snapshot root added.** The orchestrator's
`snapshot_current_tree` writes pre-destructive copies of the install
tree to `${CAVECMS_SNAPSHOT_ROOT:-/var/lib/cavecms/snapshots}`. The
system-default path requires the runtime user to be in the
`cavecmsstate` supplementary group, which a stale PM2 daemon doesn't
have (groups load at process start). The CLI now writes
`CAVECMS_SNAPSHOT_ROOT=<install>/.cavecms-state/snapshots` into
`env.production`, provisions the dir at install time as the runtime
user, and the apply route forwards the env var to the detached
orchestrator. Step 2 (snapshot) now succeeds on every fresh CLI
install — no daemon restart needed.

**Healthz preflight URL derived from PORT (was: 0.1.26 unreleased).**
The orchestrator's `/healthz` probe defaulted to `127.0.0.1:3040`
(the bare-metal port). CLI installs run on whatever port the operator
chose (PORT in env.production). The apply route now derives
`CAVECMS_HEALTHZ_URL=http://127.0.0.1:${PORT}/healthz` if not
explicitly set, so preflight finds the actual running app on the
first try.

### What this looks like for customers

A 0.1.27+ CLI install applies subsequent updates with no manual
steps. The 6-step modal walks cleanly:

1. **Getting ready** — healthz probe against the install's actual port.
2. **Downloading the new version** — tarball + SHA-256 + Ed25519 verify.
3. **Preparing your data** — `install-migrate.mjs`, not pnpm.
4. **Building your site** — no-op (tarball is pre-built); just re-link
   the standalone static + public symlinks.
5. **Switching to the new version** — atomic swap + pm2 reload.
6. **Verifying the new version** — healthz with the new commit SHA.

If any step fails, the orchestrator restores the snapshot taken at
step 2 and reloads pm2 — rollback is unchanged.

### Migration impact

**Customers on 0.1.16–0.1.26** can't self-update past these
orchestrator bugs (the broken orchestrator is bundled in their
standalone). They need a fresh `npx create-cavecms@latest` re-install
to pick up the 0.1.27 orchestrator. Subsequent updates work in-app.

**Customers on the bare-metal deploy.sh flow** are unaffected — the
git-mode branch keeps the original `pnpm install` / `pnpm db:migrate`
/ `pnpm build` path verbatim.

### Provenance

This release replaces four in-flight commits and follows a full audit
of `scripts/cavecms-update.sh`, `scripts/lib/cavecms-update-helpers.sh`,
`scripts/cavecms-watchdog.sh`, `app/api/admin/updates/apply/route.ts`,
`lib/updates/detachMechanism.ts`, `packages/create-cavecms/bin/create-cavecms.mjs`,
and the live state of an affected install — so every code path needed
by the updater on CLI installs is wired against the same set of env
vars + paths the CLI actually provisions.
