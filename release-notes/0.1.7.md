## 0.1.7

Fix the runtime: 0.1.6 reached step 7 (PM2 start) for the first time
and the started process immediately errored with
`[start-standalone] .next/standalone/server.js not found`.

### What was broken

`build-zip.mjs` was flattening the Next.js standalone output to the
install root (`dst: '.'` in the manifest), so `.next/standalone/server.js`
ended up at `<install>/server.js`. But every customer-side script
that touches the bundle expects the canonical `.next/standalone/`
subpath:

- `scripts/start-standalone.mjs` (line 26): `serverJs = join(cwd, '.next', 'standalone', 'server.js')`
- `scripts/install-migrate.mjs` (line 49): `STANDALONE_PKG = join(INSTALL_ROOT, '.next', 'standalone', 'package.json')`
- cPanel `app.js` shim emitted by the CLI: `require('./.next/standalone/server.js')`

All three were written against the dev-repo layout where `pnpm build`
naturally produces `.next/standalone/server.js`. The flatten was a
silent break: install-migrate compensated via its fallback createRequire
on import.meta.url, but start-standalone had no fallback and PM2
restart-looped 9 times before pausing.

### Fix

`build-zip.mjs` standalone manifest entry changed from `dst: '.'` to
`dst: '.next/standalone'`. The release zip now mirrors the dev-repo
layout exactly — `<install>/.next/standalone/server.js`,
`<install>/.next/standalone/node_modules/...`,
`<install>/.next/standalone/.next/...`. Every script's hardcoded path
now works.

### Why this cascade took 7 releases

Each 0.1.x release exposed the next layer:
- 0.1.1: signing pipeline + CAVECMS_COMMIT (Session A's work)
- 0.1.2: mysql2 + drizzle-orm need to be in `serverExternalPackages`
- 0.1.3: pnpm symlinks in staging (`cp -a` not `fs.cpSync`)
- 0.1.4: BSD `zip -y` to preserve symlinks in the archive
- 0.1.5: backfill the migration journal with orphan .sql files
- 0.1.6: install-migrate `multipleStatements: true` for hand-written SQL
- 0.1.7: standalone bundle at `.next/standalone/`, not install root

Every layer was the same root cause — assumptions about file layout
that the dev-repo satisfied but the release zip silently violated.
The customer-journey test caught all of them in one chain.

### Operator impact

Fresh `npx create-cavecms` installs now reach step 7 cleanly AND the
started process actually serves on its port. Existing 0.1.6 installs
will pick up 0.1.7 via the in-app updater, BUT the in-app updater
expects a `.next/standalone/server.js` it'll restore via snapshot —
operators on 0.1.6 should re-install fresh rather than trying to
update over a broken-layout install.
