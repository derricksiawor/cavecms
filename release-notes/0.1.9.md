## 0.1.9

Sync `db/schema-fingerprint.txt` to what a fresh post-migration DB
actually computes to. 0.1.8 wired install-migrate to populate the
`schema_fingerprint` row, but the bundled file still carried a stale
dev-snapshot value that didn't match the post-migration state. Every
fresh boot fataled with `schema_fingerprint_mismatch`.

### What was broken

The bundled `db/schema-fingerprint.txt` was computed at some point in
the dev cycle and never refreshed after 10+ hand-written migrations
(0011-0022) landed. Reflected an out-of-date schema snapshot — not
what `info_schema.COLUMNS` reports after applying the canonical
0000-0022 migration sequence against a fresh DB.

Install-migrate (as of 0.1.8) computes the LIVE post-migration
fingerprint and writes it to the row. Boot then compares row vs file
and fatals if they differ. They differed — file was wrong.

### Fix

Replaced `db/schema-fingerprint.txt` with the canonical post-migration
value computed against a fresh DB on MariaDB 10.6.22:

```
04e7b51359de589b8c68d2a9cc1359f00fd892f68255bf0e8c12fcfd54a8cd51
```

### Known limitation

`information_schema.COLUMNS` output can vary across MariaDB versions
(`now()` vs `current_timestamp()` in `column_default`, `DEFAULT_GENERATED`
prefix in `extra`, etc.). The bundled fingerprint reflects MariaDB
10.6.22's output. Customers on a substantially different MariaDB
version may still hit `schema_fingerprint_mismatch`. The proper
long-term fix is to canonicalize the fingerprint input to be
version-agnostic (strip version-dependent EXTRA prefixes, normalize
`now()` vs `current_timestamp()`); for 0.1.x we ship the 10.6.22
value and document the limitation.

### Follow-up worth doing later

1. Pre-canonicalize `column_default` + `extra` in update-fingerprint
   and install-migrate so they produce identical fingerprints across
   MariaDB versions.
2. Add CI step that runs `pnpm db:fingerprint` + verifies
   `db/schema-fingerprint.txt` is in sync with the migration output.
   Catches stale-file regression before it ships.

### Eight CMS releases + two CLI releases later

The 0.1.1 cascade closes. Fresh installs via `npx create-cavecms` on
MariaDB 10.6.22 should now reach a serving Node process on the
configured port.
