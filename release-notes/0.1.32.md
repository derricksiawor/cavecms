## 0.1.32

In-app updater hotfix — archive-format mismatch.

### Fixed

**Orchestrator extracted release archives with `tar -xzf` but the
publisher ships ZIP files.** `extract_tarball_atomic` and
`verify_tarball` were written assuming GitHub-style `.tar.gz`
tarballs. The actual release pipeline produces `.zip` archives
(matching what `npx create-cavecms` consumes at fresh-install time).
Every in-app update on a CLI install hit
`gzip: stdin has more than one entry--rest ignored` and failed at
step 2 with the generic "Something went wrong" trap message
(specific extract-failure status was overwritten by the unhandled-
exit trap when the extract returned non-zero AND the subsequent
rollback path ran into its own non-zero exit).

Both functions now sniff the archive's 4-byte magic header (PK
\x03\x04 for ZIP, 1F 8B for gzip) and branch to the correct tool:
`unzip` for ZIP, `tar -tzf` / `tar -xzf` for tar.gz. Path-traversal
+ symlink-rejection safety checks have matching ZIP-aware probes
(`unzip -l` for entries, `unzip -Z` for symlink detection).

### Migration impact

**Critical**: customers on installs from 0.1.16-0.1.31 can't
self-upgrade past this bug — their on-disk orchestrator has the
broken `tar -xzf` call. They need a fresh `npx create-cavecms@latest`
re-install to pick up the 0.1.32 orchestrator. Subsequent updates
work in-app.
