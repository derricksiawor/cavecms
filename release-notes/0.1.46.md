## 0.1.46

Hotfix on top of 0.1.45 — fresh installs from `npx create-cavecms@latest`
crashed at boot with `schema_fingerprint_mismatch`.

### Fixed

**Schema fingerprint reflected dev-DB state, not fresh-install state.**
`db/schema-fingerprint.txt` was baked against the publisher's local
dev DB, which had drifted from a true fresh-migration state (likely
from earlier hot-patches to repair the deleted_at column defaults
during the 0023 work). Customers installing 0.1.45 saw every
migration run cleanly, then the boot-time fingerprint check refused
to start with:

```
{"level":"fatal","msg":"schema_fingerprint_mismatch",
 "expected":"35ca456723cb...", "actual":"4aee206b4adf..."}
```

Migration 0024 is data-only (UPDATEs / DELETEs / INSERTs on
`content_blocks` + `audit_log` — no `ALTER TABLE`), so the fingerprint
should NOT have differed between 0023 and 0024. The mismatch
identifies a stale baseline, not a runtime divergence.

The new baseline `4aee206b...` is the fingerprint a clean install
produces after applying all migrations 0000-0024 from scratch
against a fresh MariaDB instance, which is the customer scenario
the gate is built to enforce. Customer installs of 0.1.46 boot
cleanly.

### Migration impact

None — schema is identical to 0.1.45, just the fingerprint baseline
was wrong. No new migrations.

### Upgrade path for 0.1.45 customers

Either:
1. Use the in-app updater to pull 0.1.46 (the rolling-update path
   re-runs `pnpm db:fingerprint` after migrations so the gate stays
   correct).
2. Or, for the manual path on a 0.1.45 install that's still booting:
   ```
   echo 4aee206b4adf3af1c0e6a19b684fa8b614e54c9ff812e1de862ae9e2a825a95a \
     > <install>/.next/standalone/db/schema-fingerprint.txt
   ```
   then restart the service.

### CLI

`create-cavecms@0.1.46` published in lockstep.
