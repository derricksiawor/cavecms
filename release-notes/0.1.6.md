## 0.1.6

Fix install-migrate to accept raw-`;`-separated migration files (not
just drizzle's `--> statement-breakpoint`-decorated ones).

### What was broken

`scripts/install-migrate.mjs` created its DB connection with
`multipleStatements: false` and split each migration on
`--> statement-breakpoint` markers. Drizzle-kit generates files with
those markers — every migration 0000-0010 has them. But 5 of the
later hand-written migrations (0011, 0012, 0013, 0014, 0022) ship
multi-statement SQL separated only by raw `;`. mysql2's prepared-
statement protocol rejects multi-statement strings, so the install
died at migration 0011 with `You have an error in your SQL syntax`.

### Fix

Set `multipleStatements: true` on the install-migrate connection. The
existing splitter still cuts drizzle-generated files at the breakpoint
markers and dispatches one statement per query (for diagnostic
precision), but marker-less files now pass through as a single
multi-statement query that MariaDB executes as a sequence.

### Why not edit the .sql files instead

Drizzle's `__drizzle_migrations` table tracks applied migrations by
sha256 of file content. Adding markers to existing .sql files would
change every dev DB's recorded hash → next migration run would see
the new hash as "not yet applied" → re-apply attempts on already-
applied schema → spurious errors. The migrator-side fix is the only
non-destructive path.

### Operator impact

- Fresh `npx create-cavecms` installs now complete migrations cleanly.
- Existing installs unaffected — the migrator change ships only in
  the release zip's `scripts/` dir, applied via in-app updates or
  fresh-install retries.

### Follow-up worth doing later

The 10 hand-written migrations (0011-0022, gaps at 0018+0020) bypassed
`drizzle-kit generate`. A linter that fails CI when a new `.sql`
file lacks a matching journal entry would catch this earlier.
