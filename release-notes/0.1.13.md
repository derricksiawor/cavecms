## 0.1.13

Closes a release-pipeline bug surfaced while preparing to test the
in-app updater on a fresh customer install.

### Fixed

**Manifest entries didn't carry the git commit SHA.** Every previous
release's manifest entry (0.1.0 → 0.1.12) shipped without a `sha`
field. `create-cavecms` reads that field at install time to stamp
`CAVECMS_COMMIT` into `env.production`; with the field missing, the
CLI fell back to the sentinel value `dev` and printed:

> Manifest entry has no `sha` field — env.production will carry
> CAVECMS_COMMIT=dev. In-app updater will refuse to run until you
> re-install from a newer CLI.

`lib/updates/getCurrentVersion.ts` then classifies a `dev` commit as
"no update possible" and `lib/updates/applyApi.ts` refuses to apply
with `cannot_apply_from_dev`. Net effect: every install built from
0.1.12 or earlier was permanently locked out of the in-app updater.

`scripts/release/build-zip.mjs` (publisher-private) now resolves the
build's `git rev-parse HEAD` and writes it as the `sha` field on the
new manifest entry. Customers installing 0.1.13+ get a real
40-char SHA in `env.production` and the updater works end-to-end.

### Migration impact

None for the codebase. Operationally: any customer running an install
built from 0.1.12 or earlier needs to either (a) re-install from
0.1.13+ to refresh `CAVECMS_COMMIT`, or (b) wait for a future
self-healing flow that backfills the SHA from a known-good manifest
entry. Re-install is the recommended path for now.
