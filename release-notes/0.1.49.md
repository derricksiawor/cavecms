## 0.1.49

Single-fix release on top of 0.1.48. Every CMS slug that wasn't a
reserved route returned 404 in production builds.

### Symptom

After installing any of the 8 templates, every page the operator
clicked through the site nav 404'd unless the URL matched a
hand-written app-route file. Visible breakdown:

- `harbor-and-lane`: home + contact rendered; /agents, /listings,
  /neighborhoods all 404'd
- `anara-wellness`: home + contact rendered; /treatments,
  /practitioners, /journal, /booking all 404'd
- Every other template: same pattern — only home + the reserved
  slugs (`/contact`, `/privacy`, `/terms`, `/blog`, `/projects`) worked
- In Safari the 404 also content-sniffed as a download because the
  empty bodyless 404 response had no `Content-Type` header

The operator's custom LOGIN_PATH still resolved correctly in Chrome
(the auth route doesn't route through the rewrite path) but Safari
behaved differently because of stale HSTS + the same 404 chain.

### Root cause

CaveCMS uses a `/_page/[slug]` internal route as the dispatch target
for non-reserved CMS slugs — middleware rewrites `/agents` →
`/_page/agents` and Next.js then renders the page from DB. The
`%5Fpage` directory-name was the URL-encoded form documented by
Next.js for routing a segment that starts with `_`.

In Next.js 15.5 standalone builds (vercel/next.js#82043 + adjacent
behaviour), the runtime dynamic-route matcher silently refuses any
URL whose first segment begins with `_` even when the routes-manifest
registers the path. The middleware rewrite committed to a URL that
the runtime then rejected, and the response was an empty 404. Safari
saw `404 + no Content-Type + chunked transfer + 5 bytes of body` and
treated it as a binary download.

### Fix

The render-route prefix moved from `/_page/...` to `/cms-render/...`
— same handler, non-underscore-prefixed path, dispatches reliably on
standalone builds. Specifically:

- `app/%5Fpage/[slug]/page.tsx` → `app/cms-render/[slug]/page.tsx`
- `middleware.ts` rewrite target + direct-access refuse switched to
  `/cms-render/`
- `lib/cms/page-slug.ts` added `cms-render` to `RESERVED` so an
  operator cannot mint a CMS page that would mask the internal prefix

The migration is transparent to operators: the URL bar still shows
`/agents` / `/listings` etc. — the rewrite target is internal and
never visible.

### Operator action

None. `npx create-cavecms@latest` for a fresh install picks 0.1.49
automatically. Existing 0.1.48 installs get the fix via the in-app
updater (Settings → Updates → Update now).

### CLI

`create-cavecms@0.1.49` published in lockstep, no functional change
from 0.1.48.
