## 0.1.11

Hotfix for 0.1.10 — the bare `/` install-gate redirect throws
`TypeError: Invalid URL` in Next 15's edge middleware runtime when the
Location header is a path-only string. 0.1.10 swapped to a relative
Location to dodge the loopback leak but tripped over Next's URL
validation, so fresh installs returned 500 on `/` before the wizard.

### Fixed

**Install-gate redirect throws on `/`.** 0.1.10's middleware emitted
`new NextResponse(null, { headers: { location: '/install' } })` to
avoid resolving against `req.url` (which contains the loopback host
because `start-standalone.mjs` binds Next to `HOSTNAME=127.0.0.1` for
defence-in-depth). Next 15 internally constructs a `URL` from the
Location header and rejects path-only strings with
`TypeError: Invalid URL`. The result was a 500 on every public path
of a fresh install until the wizard completed.

Proper fix: build the redirect URL from the forwarded `Host` +
`X-Forwarded-Proto` headers that nginx sets. That gives an absolute
URL pinned to the public hostname the visitor actually typed,
regardless of how the upstream is bound. Falls back to `req.nextUrl`
host / protocol when the headers are absent (direct-to-Node access,
non-proxy deploys).

### Migration impact

None. Middleware-only change. Re-deploy / re-install picks it up.
