## 0.2.5

Built for shared hosting's memory limits: CaveCMS now fits itself inside your hosting plan's cap instead of getting silently killed by it, and the command-line updater finally talks to you.

### Fixed — no more mystery 503s on memory-capped hosting

Shared hosting (cPanel/CloudLinux) caps how much memory your whole account can use. Node didn't know about that cap, so under pressure it kept allocating until the server killed it — your site went down with nothing in any log. Now:

- The installer tunes the app to your account's actual memory limit at install time: it caps Node's memory appetite to fit (so under pressure the site gets briefly slower instead of dying) and trims the number of app workers the host spawns.
- The background update pre-download — the thing most likely to tip a tight account over — now verifies the downloaded file piece by piece instead of holding all ~150 MB in memory, and it steps aside entirely when memory is running low. The download simply happens when you click Update instead.
- The app records its effective memory ceiling at startup, so "is the tuning active?" is a one-line check.

### New — see your memory budget

Settings → Updates now shows the memory your hosting account actually has, with a plain note when it's on the tight side and what to ask your host for. No more guessing why things went quiet.

### Improved — the command-line updater keeps you posted

`cavecms update` now narrates each step as it runs — getting ready, downloading, preparing your data, and so on — and ends with a clear verdict: what succeeded, or exactly which step failed and where its log is. Previously it could finish (or fail) without printing a word.

On cPanel it also figures out your site's public address by itself for the health checks, the same way the dashboard updater does — no special environment variables needed.
