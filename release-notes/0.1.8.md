## 0.1.8

Have install-migrate populate the `schema_fingerprint` row so the
standalone server boots after a CLI install.

### What was broken

`instrumentation.ts` at boot reads `db/schema-fingerprint.txt` (baked
into the release zip at build time) and compares against a row in the
`schema_fingerprint` table. In dev / deploy.sh flows that row gets
populated by `pnpm db:fingerprint` (which runs
`scripts/update-fingerprint.ts` after `pnpm db:migrate`). The CLI's
install-migrate had no equivalent — migrations applied cleanly, the
table was created, but the row was never inserted. Boot logged
`schema_fingerprint row missing; run pnpm db:fingerprint` and exited 1
in prod mode. PM2 restart-looped 8 times before pausing.

### Fix

`scripts/install-migrate.mjs` now runs `updateSchemaFingerprint(conn)`
at the end of `main()`, after all migrations apply + before releasing
the advisory lock. The function ports `update-fingerprint.ts`'s
algorithm to pure JS:

- SELECTs from `information_schema.COLUMNS` for the 22 tracked tables
  (same list as `TRACKED_TABLES` in update-fingerprint.ts, kept in
  sync by comment).
- Canonicalizes each row to the exact line shape boot expects:
  `<table>.<column>:<type>|null=<Y/N>|default=<D>|key=<K>|extra=<E>`.
- SHA-256 over the joined canonical string.
- `INSERT … ON DUPLICATE KEY UPDATE` into the single-row table.

Computes against the live post-migration schema (not the bundled
file), so a future migration that drifts from schema.ts still surfaces
at boot via the file/row mismatch — defense-in-depth preserved.

### Operator impact

Fresh `npx create-cavecms` installs now boot cleanly. Existing 0.1.7
installs that already have the schema applied AND have the
schema_fingerprint row populated by other means (e.g. an operator who
ran `pnpm db:fingerprint` manually) are fine — install-migrate's
INSERT is idempotent via ON DUPLICATE KEY UPDATE.
